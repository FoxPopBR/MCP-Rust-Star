# Estudo de Caso: GSD-2 (Get Shit Done - Evolution)

## Visão Geral
O **GSD-2** é a evolução de um framework de prompts que se transformou em um sistema autônomo de "Coding Agent". Ele foi projetado para permitir que agentes trabalhem por longos períodos de forma autônoma, mantendo a visão sistêmica e sem "se perder" no meio do processo.

## Inovação Tecnológica: Pi SDK e RTK
Diferente da versão 1, que dependia apenas de prompts, o GSD-2 é uma CLI construída sobre o **Pi SDK** (TypeScript). Isso permite controle direto sobre:
-   **Limpeza de Contexto**: Resetar a janela de contexto entre tarefas para evitar alucinações.
-   **Injeção de Arquivos**: Injetar exatamente o que é necessário no momento do dispatch.
-   **Gerenciamento de Estado**: Uso de um banco de dados SQLite (`gsd.db`) para persistência de progresso e recuperação de falhas.
-   **RTK (Runtime Toolkit)**: Compressão de saída de shell para economizar tokens em logs extensos.

## Arquitetura de Memória (ADR-013)
O GSD-2 implementa uma mudança fundamental na forma como o agente lembra das coisas:
-   **Memories Table**: As decisões e lições aprendidas são armazenadas em uma tabela de memória no banco de dados.
-   **Projeções de Markdown**: Os arquivos `DECISIONS.md` e `KNOWLEDGE.md` são meras "projeções" (views) dessa tabela.
-   **Evita Divergência**: Garante que o agente sempre leia da fonte de verdade (DB) e não de arquivos markdown que podem estar desatualizados.

## Mecanismos de Autonomia e Confiabilidade
-   **Single-Writer Engine v3**: Um kernel de orquestração que garante mutações de estado atômicas (só escreve no jornal após a validação do passo).
-   **Detecção de Stuck-Loops**: O orquestrador detecta quando o agente está repetindo os mesmos comandos e intervém.
-   **Recuperação de Erros 5xx**: Se o provedor de LLM falha (erro 500), o sistema pausa e retenta em vez de abortar o progresso.
-   **Auto-Commit**: Gerenciamento automático de branches e commits git baseados no progresso das tarefas.

## Comandos Críticos de Governança
-   **/gsd verdict**: Permite que o humano (ou outro agente) dê um veredito manual de "pass", "needs-attention" ou "needs-remediation" para desbloquear um milestone.
-   **/gsd brief**: Gera briefs visuais em HTML (diagramas, planos, diffs) para que o humano possa revisar o trabalho rapidamente.
-   **/gsd doctor**: Ferramenta de diagnóstico que verifica a integridade do workspace e conflitos de merge antes de prosseguir.

## Filosofia de Execução
O GSD-2 segue a regra de **"Read-required-before-write"**. O agente é proibido de escrever em um arquivo que ele não leu na sessão atual, reduzindo drasticamente as chances de sobrescrever código importante por falta de contexto.
